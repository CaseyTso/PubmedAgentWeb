01 / Overview

系统概述

问题

在 CAR-T 细胞治疗领域，CRS 相关文献每天都在 PubMed 上更新。研究人员需要持续追踪新发表的论文以了解最新进展，但手动检索、阅读摘要、筛选相关文献是一项耗时且容易遗漏的重复性工作。

▸ 每天新增论文 50-100 篇相关领域
▸ 手动筛选耗时 30-60 min/天
▸ 容易遗漏重要进展

解决方案

构建一个自动化 Agent，通过 NCBI E-utilities API 获取 PubMed 最新文献，利用 LLM 阅读摘要并智能评分，筛选出与研究方向高度相关的论文，通过飞书/企业微信 Webhook 实时推送至团队群聊。

▸ 全自动每日监控，零人工干预
▸ LLM 理解语义，筛选精度高
▸ 实时推送至工作群，不错过重要论文

03 / Module Design

模块详细设计

PubMed Client

src/pubmed_client.py

基于 NCBI E-utilities，采用 ESearch → History Server → EFetch 两步管道模式。 使用 reldate=2 + datetype=edat 检索最近 2 天索引的文章（多取 1 天用于避免边界遗漏）。

Step 1 — ESearch 获取 PMID 列表

# 请求参数
GET eutils/esearch.fcgi
  db=pubmed
  term=(CAR-T OR "chimeric antigen receptor") AND ("cytokine release syndrome" OR CRS)
  reldate=2  datetype=edat
  usehistory=y  retmode=json

# 响应 → webenv, query_key, count

Step 2 — EFetch 批量获取全文元数据

# 使用 History Server 批量拉取（每批 200 条）
GET eutils/efetch.fcgi
  query_key={query_key}  WebEnv={webenv}
  retstart=0  retmax=200  retmode=xml

# XML 解析 → Article(pmid, title, abstract, authors, journal, doi, keywords)

Rate Limiting 速率控制

有 API Key: 10 req/s (delay=0.11s) ｜ 无 Key: 3 req/s (delay=0.35s)

关键设计决策：使用 edat（Entrez Date，即 PubMed 收录日期）而非 pdat（发表日期），因为论文可能在发表数周后才被 PubMed 索引。reldate=2 提供一天的重叠窗口，配合 SQLite 去重避免遗漏。

Relevance Scorer

src/scorer.py

将研究者画像 (research profile) 和论文摘要一起发送给 LLM，要求以结构化 JSON 输出相关度评分和理由。

Prompt 结构

System: 你是生物医学研究助手…
  评分标准 0-10 (含描述)
  输出格式: {"score": int, "reason": str}

User:
  ## 研究者画像
  {research_description}
  ## 待评估文章
  Title: ... Abstract: ...

评分标准

0-3 边缘相关，仅涉及部分关键词
4-5 中等相关，主题重叠但非直接有用
6-7 高度相关，直接涉及研究方向
8-10 核心论文，直接回答关键研究问题

容错设计：无摘要的文章自动跳过 LLM 评分（score=0）。JSON 解析失败时返回 score=0 而非抛出异常。摘要超过 3000 字符自动截断以控制 token 用量。

Notification Module

src/notifier.py

飞书 Feishu
  • ▸消息类型: interactive (Card Schema 2.0)
  • ▸卡片 Header + Markdown Body
  • ▸请求体限制: 20 KB
  • ▸频率限制: 5 req/s, 100 req/min
  • ▸可选: HMAC-SHA256 签名验证
企业微信 WeCom
  • ▸消息类型: markdown
  • ▸支持加粗、链接、列表、引用
  • ▸内容限制: 4096 bytes (UTF-8)
  • ▸频率限制: 20 req/min
  • ▸超长消息自动截断 + 溢出提示

工厂模式：create_notifier(config) 根据配置自动创建对应的通知器实例列表，支持同时推送多个渠道（platform: "both"）。

04 / Comparison

技术选型对比

筛选策略对比：关键词匹配 vs LLM 语义评分

综合准确率、成本、灵活性三个维度评估

关键词匹配

基于布尔查询和 MeSH 词表过滤。速度快、零成本，但无法理解语义——无法区分 "CRS as an adverse event" 和 "CRS pathogenesis mechanism"，容易产生大量误报。

LLM 语义评分 ✓

理解自然语言描述的研究兴趣，阅读摘要后做出上下文感知的判断。能区分"仅提及 CRS"和"深入研究 CRS 机制"。成本极低 ($0.01-0.05/天)，是本系统的选择。

通知方式对比

Email vs Webhook 推送

维度 Email Webhook (飞书/企微) ✓
实时性 可能进垃圾箱，延迟不定 秒级送达，群聊实时可见
富文本 HTML 模板兼容性差异大 Markdown / 卡片原生支持
协作性 单人接收，难以讨论 群聊推送，可即时讨论
配置复杂度 需 SMTP 配置，可能需密码 一个 URL 即可，无需认证
成本 免费 (但 SMTP 限额) 完全免费

日期策略对比

reldate vs mindate/maxdate

reldate=2 + edat ✓

  • ▸ 相对日期，无需计算昨天的具体日期
  • ▸ 2 天窗口提供 1 天重叠，避免边界遗漏
  • ▸ 配合 SQLite 去重，重叠不会产生重复推送
  • ▸ 对时区和服务器时间偏差有天然容错

mindate/maxdate

  • ▸ 绝对日期，需每次计算具体日期字符串
  • ▸ 精确但无重叠，边界条目可能遗漏
  • ▸ 需处理时区转换 (NCBI 使用 US Eastern)
  • ▸ 适合回补历史数据的场景

06 / Deployment

部署方案

本地 Cron

最简单
  • ▸系统 crontab 直接调度
  • ▸无额外依赖，开箱即用
  • ▸适合个人使用
# crontab -e
0 8 * * * cd /app && python main.py -c config.yaml

Docker 容器

推荐
  • ▸内置 APScheduler 守护进程
  • ▸环境隔离，可移植性强
  • ▸挂载 volume 持久化 SQLite
docker build -t pubmed-agent .
docker run -d \
  -v ./data:/app/data \
  -v ./config.yaml:/app/config.yaml \
  pubmed-agent

Serverless

云端
  • ▸AWS Lambda / 腾讯云 SCF
  • ▸按调用付费，闲时零成本
  • ▸需云端数据库替代 SQLite
# 触发器: CloudWatch Events
cron(0 0 * * ? *)
# UTC 00:00 = 北京 08:00

08 / Quick Start

使用指南

1

安装依赖

pip install -r requirements.txt
2

配置文件

cp config.example.yaml config.yaml
vim config.yaml   # 填入你的 API Keys

需要填写的关键配置：

pubmed.api_key

NCBI 账户获取 (可选)

llm.api_key

OpenAI / DeepSeek Key

notify.feishu.webhook_url

飞书群聊机器人 URL

3

试运行 (Dry Run)

python main.py --config config.yaml --dry-run

Dry-run 模式会执行完整的检索 + 评分流程，但不发送 webhook 通知，仅在终端输出筛选结果。用于验证配置和查询是否正确。

4

启动定时服务

# 方式 A: 守护进程模式 (内置 APScheduler)
python main.py --config config.yaml --daemon

# 方式 B: Docker (推荐)
docker build -t pubmed-agent .
docker run -d --name pubmed-agent \
  -v $(pwd)/data:/app/data \
  -v $(pwd)/config.yaml:/app/config.yaml \
  --restart unless-stopped \
  pubmed-agent

默认每天早上 8:00 (Asia/Shanghai) 自动运行。可在 config.yaml 的 schedule.cron 中自定义。
